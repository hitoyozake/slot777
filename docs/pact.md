# PACT分析

PACT（People / Activities / Context / Technologies）フレームワークを用いて、ユーザーとその利用状況を整理します。

---

## P — People（ユーザー像）

### ペルソナ1: 子育て中の保護者（メインターゲット）
- 年齢: 25〜40代
- くるくるチャイムを子どもへのプレゼントとして購入済み
- 子どもと一緒に遊ぶ時間を楽しみたいと考えている
- スマートフォンはほぼ毎日使用するが、ゲームは普段あまりしない
- 複雑な操作や長い説明は読まない
- **期待:** 子どもがチャイムを鳴らすたびに画面が反応し、一体感のある遊び体験

### ペルソナ2: 幼児〜小学生の子ども（間接ユーザー）
- 年齢: 2〜8歳
- くるくるチャイムでボールを転がすこと自体が楽しい
- 画面に変化があると喜ぶ
- スロットのルールは理解していなくてよい
- **期待:** ボールを入れると画面がキラキラして何かが起きる体験

### ペルソナ3: カジュアルゲームプレイヤー（サブターゲット）
- 年齢: 20〜40代
- スロットやゲームセンターに親しみがある
- フィーバーゲージやボーナスといったゲームメカニクスを理解している
- くるくるチャイムを持っていない可能性があるため、シミュレータも利用
- **期待:** 連打的にベルを鳴らしてフィーバーを狙う攻略的な楽しみ方

## A — Activities（ユーザーの行動）

### 物理的な活動
| 活動 | 説明 |
|------|------|
| ボールを投入する | くるくるチャイムの穴にボールを入れる |
| 連続投入する | フィーバーを維持・上昇させるために素早く投入し続ける |
| チャイムを聞く | ベルの音でゲームの反応を確認する |

### デジタルの活動
| 活動 | 説明 |
|------|------|
| スロットを回す | SPIN ボタンをタップ・クリックする |
| BETを選ぶ | 賭けるクレジット量を選択する |
| フィーバーゲージを見る | 現在のゲージ量・残り時間を確認する |
| 当たり演出を楽しむ | 当選ラインのアニメーションを見る |
| マイクを許可する | 初回起動時にパーミッションを承認する |

### ゲームループ
```
ボール投入 → ベル音 → ゲージ上昇 → スロットを回す → 当否確認 → （繰り返し）
                ↓
           ゲージ100% → フィーバー！→ 高確率スロット → WIN演出
```

## C — Context（利用文脈）

### 物理的な環境
| 項目 | 内容 |
|------|------|
| 場所 | 自宅のリビング・子ども部屋 |
| くるくるチャイムの位置 | 机やソファのそば、スマホと並べて置く |
| 騒音環境 | テレビ・BGM・子どもの声などの環境音がある |
| 照明 | 昼間の自然光〜夜の室内照明（画面視認性に影響） |

### 社会的な文脈
- 親子で一緒に遊ぶシーンがほとんど
- 子どもが主導してボールを入れ、親がスマホを操作するスタイルが想定される
- ゲームの勝ち負けよりも「反応が起きること」が楽しさの本質

### 技術的な制約
| 制約 | 対応方針 |
|------|----------|
| 環境音によるベル誤検知 | 閾値調整 + クールダウンで対処、誤検知はゲーム上許容する |
| マイク許可ダイアログへの戸惑い | 許可手順をUI上で案内する |
| iOSのAudioContext制限 | タップ操作後にAudioContextを初期化する |
| マイクがない環境 | シミュレータボタンで代替 |

## T — Technologies（利用技術）

### エンドユーザーの利用デバイス
| デバイス | ブラウザ | 優先度 |
|----------|----------|--------|
| iPhone（iOS 15+） | Safari | 高 |
| Android スマートフォン | Chrome | 高 |
| Windows PC | Chrome / Edge | 中 |
| Mac | Chrome / Safari | 中 |

### 利用するWeb標準API
| API | 用途 |
|-----|------|
| Web Audio API（AudioContext, AnalyserNode） | マイク入力・周波数解析 |
| MediaDevices.getUserMedia | マイクパーミッション取得 |
| CSS Animation / Canvas | リール・エフェクト描画 |
| Web Vibration API | ベル検出時のスマホ振動フィードバック（任意） |
| Service Worker / Web App Manifest | PWA対応（オフライン・ホーム画面追加） |

### インフラ・ビルド
| 項目 | 選定 |
|------|------|
| ホスティング | GitHub Pages / Vercel（静的） |
| ビルドツール | Vite（モジュールバンドル・開発サーバー） |
| パッケージ管理 | npm |
| テスト | Vitest（ロジックユニットテスト） |

## PACT から導かれる設計原則

1. **即時フィードバック優先** — ベル検出から画面反応まで 100ms 以内を目標にする（子どもが因果関係を感じ取れるように）
2. **シンプルなUI** — 操作は「回す」「ベルを鳴らす」の2つに集約。チュートリアル不要なレベルを目指す
3. **音なし環境への配慮** — SE・BGMはデフォルトON、ミュートボタンを目立つ位置に配置
4. **縦向き（ポートレート）レイアウト優先** — 子どもがスマホを縦に持つことが多い
5. **マイク非許可でも遊べる** — コア体験をシミュレータで完結できるようにし、マイクは「より楽しくなる」オプションとして位置づける
